  live. Field names and shapes may change before launch.
</Info>

A **verification** is a reusable configuration — your redirect URL, an attached [asset](/v3/assets), and the rules a person must satisfy. Create it **once**, then reuse its link for everyone you verify. All requests use bearer [authentication](/v3/authentication).

<Note>
  This is build-time setup — where you wire VerifyYou into your product, and
  where a platform configures verification for each company it serves. Every
  integration creates at least one verification; platforms create many and
  bind a per-company [asset](/v3/assets). The live, per-user side lives in
  [Confirmations](/v3/confirmations).
</Note>

## Create a verification

Define your rules and branding once. The returned `verification_url` is the link you share with everyone.

```http
POST /v3/verification/create
Authorization: Bearer YOUR_API_KEY
Content-Type: application/json
```

```json Body
{
  "redirect_url": "https://yourapp.com/verified",
  "asset_id": "ast_8f3a29c1",
  "config": {
    "checks": {
      "mfa": "phone",
      "geo": { "bounds": [{ "lat": 29.42, "lng": -98.49, "radius": 50000 }] },
      "demographics": {
        "age": [{ "min": 25, "max": 34 }],
        "apparent_gender": ["female"]
      }
    },
    "consumable": true,
    "signals": ["device"]
  }
}
```

<ParamField body="redirect_url" type="string" required>
  Where we return the user after verification. We append `vy_confirmation` and `vy_status` as query parameters.
</ParamField>

<ParamField body="asset_id" type="string">
  A white-label [asset](/v3/assets) to brand the hosted flow. Platform feature.
</ParamField>

<ParamField body="external_tenant_id" type="string">
  Your own tenant identifier, for grouping verifications under one of the companies you serve. Duplicable. Platform feature.
</ParamField>

<ParamField body="config" type="object">
  The rules a person must satisfy and how the verification behaves. See [Config](#config).
</ParamField>

```json Response
{
  "id": "ver_3b1f9a20",
  "verification_url": "https://verifyyou.com/v/ver_3b1f9a20"
}
```

### Config

A verification starts from a **base** check — liveness and uniqueness, always on. `config` layers extra requirements on top and sets how the result behaves:

- **`checks`** — the verification types you require. The set you turn on *is* the kind of verification: phone/email MFA, a location gate, an age/sex gate. A person must pass every check you enable.
- **Behavior** — `consumable`, `allow_anonymous`, `is_sandbox`.
- **`signals`** — attributes we only *estimate and report*, never gate on.

Everything is optional; omit a check to skip it.

<ParamField body="checks" type="object">
  The requirements layered on top of the base liveness + uniqueness check.

  <Expandable title="checks">
    <ParamField body="mfa" type="string">
      Require a second factor: `"email"` or `"phone"`.
    </ParamField>

    <ParamField body="geo" type="object">
      Require the user to be physically inside `bounds` — a list of `{ lat, lng, radius }` (radius in meters). `ask_device` requests device location. To report location without enforcing it, add `"geo"` to `signals` instead.
    </ParamField>

    <ParamField body="demographics" type="object">
      Require the user's estimated demographics to match. Each field is a list of allowed values; the person must match one entry in every field you set.

      <Expandable title="demographics">
        <ParamField body="age" type="object[]">
          Allowed age ranges, each `{ "min": <int>, "max": <int> }` (inclusive). The person's estimated age must fall in one of them.
        </ParamField>

        <ParamField body="apparent_gender" type="string[]">
          Allowed values: `"male"`, `"female"`. An estimate of *apparent* gender from facial analysis — a perceived attribute, not a claim about identity. List both to accept either.
        </ParamField>
      </Expandable>
    </ParamField>
  </Expandable>
</ParamField>

<ParamField body="consumable" type="boolean" default="false">
  Make each verified human a one-time pass. After you [consume](/v3/confirmations#consume-a-confirmation) their confirmation, they're permanently barred from this verification.
</ParamField>

<ParamField body="allow_anonymous" type="boolean" default="false">
  Allow people to verify without linking an account. Anonymous verifications dedupe against the monthly-reset pool — see [Uniqueness](/v3/uniqueness).
</ParamField>

<ParamField body="is_sandbox" type="boolean" default="false">
  Run against the sandbox — test phone numbers, no real SMS or charges.
</ParamField>

<ParamField body="signals" type="string[]" default="[]">
  Attributes to estimate for everyone who comes through, surfaced as per-verification analytics in the upcoming Stats API. Signals are observations, never gates — listing one never blocks anyone or changes the flow. Available: `age`, `apparent_gender`, `device` (platform + form factor), `geo` (where they verified from).
</ParamField>

<Note>
  The same attribute can gate or only report, depending on where it sits.
  `checks.geo` requires a location; `"geo"` in `signals` just reports one.
  `checks.demographics` blocks anyone outside your age/gender selection, while
  `"age"` or `"apparent_gender"` in `signals` only estimates them for your
  stats. Where it lives in `config` tells you whether it gates or reports.
</Note>

## Retrieve a verification

```http
GET /v3/verification/{id}
Authorization: Bearer YOUR_API_KEY
```

```json Response
{
  "id": "ver_3b1f9a20",
  "redirect_url": "https://yourapp.com/verified",
  "asset_id": "ast_8f3a29c1",
  "config": {
    "checks": {
      "mfa": "phone",
      "demographics": {
        "age": [{ "min": 25, "max": 34 }],
        "apparent_gender": ["female"]
      }
    },
    "consumable": true,
    "signals": ["device"]
  }
}
```

## Update a verification

Send only the fields you want to change. Updates apply to every future use of the link.

```http
PUT /v3/verification/{id}
Authorization: Bearer YOUR_API_KEY
Content-Type: application/json
```

```json Body
{
  "redirect_url": "https://yourapp.com/done",
  "config": { "consumable": false }
}
```

## Delete a verification

Retires the link. Existing confirmations are unaffected; the link stops accepting new users.

```http
DELETE /v3/verification/{id}
Authorization: Bearer YOUR_API_KEY
```

## Enroll a user

<Note>
  Optional — for slotting verification into an existing user flow. Skip it if you just share the link.
</Note>

Enroll a specific user before they verify. Pass data you already hold and we link the verification to their VerifyYou identity, so a verified user's status **travels with them** across your integrations. Returns a personalized one-time link.

```http
POST /v3/verification/{id}/enroll
Authorization: Bearer YOUR_API_KEY
Content-Type: application/json
```

```json Body
{
  "external_id": "user_8f3a29c1",
  "phone": "+12105550142",
  "email": "user@example.com"
}
```

<ParamField body="external_id" type="string">
  Your identifier for this user. If omitted, we generate one. See [External IDs](/v2/external-ids).
</ParamField>

<ParamField body="phone" type="string">
  A pre-verified E.164 number — skips the phone-OTP step on the returned link.
</ParamField>

<ParamField body="email" type="string">
  Used when `config.mfa` is `"email"`.
</ParamField>

```json Response
{
  "verification_url": "https://verifyyou.com/v/ver_3b1f9a20?c=cnf_7a2e51bd",
  "confirmation": "cnf_7a2e51bd",
  "external_id": "user_8f3a29c1"
}
```
